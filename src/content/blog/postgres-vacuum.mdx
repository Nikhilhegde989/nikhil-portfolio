If you've seen a query jump from 10ms to 3 seconds with no code change, VACUUM is probably involved. Here's what I learned after debugging a painful production slowdown.

## Why Dead Tuples Exist

PostgreSQL uses **MVCC** (Multi-Version Concurrency Control) to handle concurrent reads and writes without blocking. Every `UPDATE` doesn't modify a row in-place — it marks the old row as "dead" and inserts a new version. Same with `DELETE`: the row isn't immediately removed, just flagged.

This is why PostgreSQL is excellent at concurrent workloads. But there's a cost: **dead tuples accumulate on disk.**

```sql
-- See dead tuple count per table
SELECT relname, n_live_tup, n_dead_tup, last_autovacuum
FROM pg_stat_user_tables
ORDER BY n_dead_tup DESC;
```

## What VACUUM Actually Does

VACUUM does two things:

1. **Marks dead tuple space as reusable** — it doesn't release memory to the OS, but marks those pages as available for new rows.
2. **Updates the visibility map** — a file that tracks which pages contain only tuples visible to all active transactions. This is critical for index-only scans.

```sql
-- Run VACUUM on a specific table
VACUUM ANALYZE orders;
```

VACUUM does **not** shrink the physical file. That's `VACUUM FULL` — and it's a heavy operation.

## VACUUM FULL: When and Why Not

`VACUUM FULL` rewrites the entire table, reclaims physical disk space, and holds an **exclusive lock** for the duration. On a 50 GB table with active traffic, this is a production outage waiting to happen.

**The rule:** Only run `VACUUM FULL` on tables that are severely bloated AND during a maintenance window. For most cases, regular `VACUUM` (or autovacuum) is the correct answer.

## Autovacuum: Your Silent Guardian

Autovacuum runs in the background and triggers VACUUM automatically. The default thresholds are conservative:

```ini
autovacuum_vacuum_scale_factor = 0.2   -- trigger when 20% of rows are dead
autovacuum_analyze_scale_factor = 0.1  -- trigger ANALYZE when 10% changed
```

For a 10M-row table, this means autovacuum won't trigger until 2M rows are dead. That's a lot of bloat before cleanup.

**Tune per-table for high-churn tables:**

```sql
ALTER TABLE order_events SET (
  autovacuum_vacuum_scale_factor = 0.01,
  autovacuum_vacuum_threshold = 1000
);
```

This makes autovacuum trigger after just 1% of rows (or 1,000 rows, whichever is larger) are dead — far more aggressive for tables that see constant writes.

## The Visibility Map and Index-Only Scans

This is the part most engineers miss. PostgreSQL maintains a **visibility map** per table — a bitmap where each bit indicates whether a page contains only tuples visible to all active transactions.

When a query can use an **index-only scan**, PostgreSQL checks the visibility map first. If the page is "all-visible," it skips the heap entirely and reads directly from the index. This is a massive performance win for read-heavy queries.

VACUUM keeps the visibility map current. Without regular VACUUM, the map goes stale, and index-only scans quietly degrade into full heap scans.

```sql
-- Check heap access vs index-only scans for a table
SELECT relname, idx_scan, idx_tup_fetch, heap_blks_read, heap_blks_hit
FROM pg_statio_user_tables
WHERE relname = 'orders';
```

## What Actually Happened in Production

We had a high-write `events` table with default autovacuum settings. Over a holiday weekend with 3× normal traffic, dead tuples accumulated faster than autovacuum could clear them. By Monday morning:

- Table bloat was sitting around 60%
- The visibility map was effectively stale
- Index-only scans had quietly degraded to heap scans
- A join query against this table went from 15 ms to 4 seconds

The fix was immediate: run `VACUUM ANALYZE events` manually, then tune per-table autovacuum settings.

```sql
VACUUM ANALYZE events;

ALTER TABLE events SET (
  autovacuum_vacuum_cost_delay = 2,
  autovacuum_vacuum_scale_factor = 0.02
);
```

## Quick Checklist

- Monitor `n_dead_tup` in `pg_stat_user_tables` regularly
- Set aggressive autovacuum thresholds on high-churn tables
- Never run `VACUUM FULL` without a maintenance window on large tables
- Check `last_autovacuum` — if it's null or weeks old, something is wrong
- Use `pg_stat_progress_vacuum` to watch an in-progress VACUUM live

VACUUM isn't glamorous. But understanding it is the difference between a midnight incident and a 30-second fix.
