The feature request sounded simple: *"When an order is processed, show the status update in the UI immediately."*

The implementation was less simple. Here's how I built a Kafka-to-WebSocket bridge in Python that handles reconnections, backpressure, and the subtle race condition that cost us a day.

## The Architecture

```
┌──────────┐    ┌─────────┐    ┌──────────────┐    ┌─────────┐
│  Worker  │───▶│  Kafka  │───▶│  FastAPI WS  │───▶│ Browser │
│ Services │    │  Topic  │    │  Consumer    │    │  Client │
└──────────┘    └─────────┘    └──────────────┘    └─────────┘
```

Workers publish events to a Kafka topic. A FastAPI service consumes that topic and maintains WebSocket connections with browser clients. Each client subscribes to updates for a specific `order_id`.

## The Connection Manager

The core is a manager that maps `order_id` → active WebSocket connections:

```python
from collections import defaultdict
from fastapi import WebSocket

class ConnectionManager:
    def __init__(self):
        self.active: dict[str, list[WebSocket]] = defaultdict(list)

    async def connect(self, order_id: str, ws: WebSocket):
        await ws.accept()
        self.active[order_id].append(ws)

    def disconnect(self, order_id: str, ws: WebSocket):
        self.active[order_id].remove(ws)
        if not self.active[order_id]:
            del self.active[order_id]

    async def broadcast(self, order_id: str, data: dict):
        dead = []
        for ws in self.active.get(order_id, []):
            try:
                await ws.send_json(data)
            except Exception:
                dead.append(ws)
        for ws in dead:
            self.disconnect(order_id, ws)
```

The `broadcast` method collects dead connections (clients that disconnected mid-flight) and cleans them up after the loop — safe iteration without modifying the list during traversal.

## The Kafka Consumer Loop

```python
from aiokafka import AIOKafkaConsumer
import asyncio, json

async def kafka_consumer_loop(manager: ConnectionManager):
    consumer = AIOKafkaConsumer(
        'order.events',
        bootstrap_servers='localhost:9092',
        group_id='ws-bridge',
        auto_offset_reset='latest',
    )
    await consumer.start()
    try:
        async for msg in consumer:
            event = json.loads(msg.value)
            order_id = event.get('order_id')
            if order_id:
                await manager.broadcast(order_id, event)
    finally:
        await consumer.stop()
```

This runs as a background task in FastAPI using `lifespan`:

```python
from contextlib import asynccontextmanager
from fastapi import FastAPI

manager = ConnectionManager()

@asynccontextmanager
async def lifespan(app: FastAPI):
    task = asyncio.create_task(kafka_consumer_loop(manager))
    yield
    task.cancel()
    await asyncio.gather(task, return_exceptions=True)

app = FastAPI(lifespan=lifespan)
```

## The Edge Case That Cost Us a Day

Everything worked in development. In production, we hit this scenario: a client connects *just after* a Kafka message is published for their order. The message was already consumed and broadcast — to zero connected clients. The client connects, fetches the current state over HTTP (gets `PROCESSING`), then receives no WebSocket update because the `SHIPPED` event already passed.

The fix: send the current state immediately on WebSocket connect, then stream future events.

```python
@app.websocket("/ws/orders/{order_id}")
async def order_ws(order_id: str, ws: WebSocket):
    await manager.connect(order_id, ws)
    # Catch-up: send current state on connect
    current = await db.get_order_state(order_id)
    await ws.send_json({"type": "state", "data": current})
    try:
        while True:
            await ws.receive_text()  # keep-alive, handles pings
    except WebSocketDisconnect:
        manager.disconnect(order_id, ws)
```

Now clients always see the current state on connect, then receive live diffs as they happen.

## Lessons Learned

**Test with slow clients.** A browser that doesn't drain the WebSocket buffer fast enough causes backpressure. Async sends queue up. Add send timeouts.

**Monitor consumer lag.** If your Kafka consumer falls behind the partition, UI updates arrive late. Lag monitoring is non-negotiable in production.

**Auto-reconnect on the client.** The browser WebSocket API does not reconnect automatically. Write a simple exponential backoff loop or use a library — your users will disconnect on mobile constantly.

**Group ID and horizontal scaling.** If you run multiple instances of the WS server, they'll compete for Kafka messages under the same group ID — each instance gets a subset of events. Add a Redis pub/sub layer between Kafka and the WS servers so every instance can relay every event to its own connected clients.

This pattern holds up well at moderate scale. For high throughput, the Redis pub/sub layer between Kafka and WebSocket servers is the next step.
